Arquitectura para desarrolladores

Monorepo — Libs, productos verticales, libs comercializables y escalado a microservicios cuando haga falta (arquitectura tipo Lego)

Audiencia: desarrollo y diseño

Objetivo

Explicar por qué la arquitectura actual (monorepo con apps + libs) es mejor para el equipo, cómo crear y mantener features como libs, cómo escala (productos verticales, libs comercializables), agnosticismo por feature (incl. módulos federados para migraciones y legacy), por qué no usar microservicios desde el principio y cómo escalar a microservicios cuando haga falta — como un Lego.

1. Por qué esta solución es mejor (para devs)

  • Una sola fuente de verdad: Tipos compartidos (shared-types) entre front y back. Sin duplicar DTOs ni desalinear contratos.
  • Contrato alineado: La app de front consume la misma API REST que expone la app de back; mismos endpoints, mismos tipos.
  • Un clon, una instalación: Un solo repo, pnpm install, y arrancar front y API desde la raíz. Sin coordinar dos repos ni versiones.
  • Features como libs: Cada dominio (auth, usuarios, objetivos, fichajes, etc.) vive en su lib. Cambios acotados, tests por lib, dependencias claras.
  • Boundaries y tags: Las dependencias entre libs están definidas (shell → features → ui/platform → shared-types). Menos acoplamiento y más refactors seguros.
  • CI/CD y cache: El CI/CD es el núcleo inicial del proyecto: un solo pipeline (build, lint, tests de integración y E2E); Nx ejecuta solo lo afectado. Menos tiempo de build y calidad automática en cada commit.
  • Módulos federados (opcional): Con Module Federation se pueden usar distintos frameworks en la misma app (React, Vue, Angular, legacy). Facilita migraciones graduales y traer proyectos heredados al monorepo sin reescribir todo de golpe.

2. Front antiguo (átomos/moléculas) vs nuevo: mejor para todos los devs

Por qué la arquitectura del front antiguo era peor

En el proyecto inicial el front se organizaba por Atomic Design (átomos, moléculas, organismos, templates) y por tipo de componente, no por dominio ni por feature:

  • Todo mezclado por “tamaño”: Carpetas atoms/, molecules/, organisms/, templates/. Un mismo dominio (ej. objetivos, usuarios, login) queda repartido: un átomo aquí, una molécula allí, un organismo en otra carpeta. No hay un solo lugar “objetivos” o “auth”; hay que saltar entre capas atómicas para entender una funcionalidad.
  • Lógica, UI y mocks revueltos: Los mocks vivían en constants/mock/; las pantallas (Home por rol, usuarios, etc.) importaban de atoms/molecules/organisms y de esos mocks. Sin separación clara entre “datos de dominio”, “componente reutilizable” y “página”. Difícil saber qué tocar para cambiar una feature.
  • Inconsistencia átomo vs molécula: ¿Un botón con icono es átomo o molécula? ¿Un formulario de login es organismo o template? La frontera es arbitraria y genera discusiones y estructura poco estable. No hay reglas claras de dependencia (quién puede importar a quién).
  • Acoplado a una sola app Next: Toda la estructura (App Router, layouts, páginas) es de Next. No hay una “lib de UI” ni “lib de feature” reutilizable; todo es parte del proyecto Next. Un dev de Angular o Vue no puede reutilizar nada sin meterse en el mundo Next.

Por qué una arquitectura “solo Next” es peor para equipos con devs Angular, Vue, etc.

  • Si todo el front es “la app Next” (App Router, Server Components, convenciones de Next), cualquier persona que venga de Angular, Vue o otro stack tiene que aprender primero Next y su modelo mental (rutas, layouts, client/server). El proyecto no es “común”; es “de Next”.
  • No hay contrato reutilizable: la UI y la lógica viven dentro del repo Next. Para que un dev Angular contribuya, tendría que tocar Next o duplicar componentes. No hay una lib de UI ni de features que cualquier framework pueda consumir.

Por qué la nueva arquitectura es mejor para todos

  • Organización por feature (dominio), no por átomo/molécula: Hay una lib por dominio (auth, usuarios, objetivos, fichajes…). Quien trabaja en “objetivos” va a libs/frontend/features/objetivos. No hay que buscar en cuatro capas atómicas; la feature está en un solo sitio.
  • UI en una lib: Componentes reutilizables (botones, inputs, cards) viven en ui / ui-layout. Cualquier app (Next, otra SPA, en el futuro Angular o Vue) puede consumir esa lib. Diseño y dev comparten el mismo sistema de componentes; un dev Angular que entre al proyecto puede trabajar en la lib de UI o en una feature sin tener que dominar Next al 100%.
  • Contrato común (shared-types, API): Lo que une a front y back (y a distintos fronts) es el contrato: tipos y endpoints. Eso es independiente del framework. Tanto un dev React/Next como uno Angular pueden leer los tipos, entender la API y contribuir a una feature o a la UI.
  • Común a todos: La arquitectura no es “para devs de Next”; es “apps + libs por feature + lib de UI + contrato”. Quien venga de Angular o Vue puede trabajar en libs (features o UI) con TypeScript y el mismo contrato; la app que orquesta puede ser Next hoy y otra cosa mañana sin tirar el trabajo de nadie.

Resumen: la estructura átomos/moléculas/organismos mezcla todo por tipo de componente y no por dominio; una arquitectura centrada solo en Next deja fuera a quien no trabaje en Next. La nueva (libs por feature + UI lib + contrato común) es mejor para cualquier dev y para equipos con perfiles diversos.

3. Back antiguo vs nuevo: por qué esta arquitectura es mejor

Por qué la arquitectura del back antiguo era peor

En el proyecto inicial el back se organizaba como API Gateway (Apollo) + subgraphs GraphQL + microservicio de auth en Express, con varios procesos y sin contrato compartido con el front:

  • GraphQL que el front no consumía: El gateway exponía un esquema GraphQL (login, usuarios, etc.), pero el front llamaba a REST con otros puertos y rutas. El contrato GraphQL no era la interfaz real entre front y back; había desalineación (p. ej. login con username en front y email en el esquema).
  • Varios procesos y puertos: Gateway en un puerto, microservicio de auth en otro. El front apuntaba por defecto a un puerto donde no corría nada; para que funcionara había que configurar varios entornos. Más despliegues, más redes, más puntos de fallo.
  • Sin tipos compartidos: No existía un paquete común (shared-types). Los DTOs y el esquema GraphQL vivían en el back; el front tenía sus propias interfaces y mocks. Cualquier cambio de contrato obligaba a tocar ambos sitios a mano y era fácil desincronizar.
  • Complejidad innecesaria para el tamaño del producto: Subgraphs, dataSources, resolvers y un microservicio solo para auth añadían complejidad operativa y de desarrollo (CORS, JWT, init duplicado) sin beneficio para un equipo pequeño y una sola app cliente.

Por qué una arquitectura “solo GraphQL/microservicios” es peor para equipos y operación

  • Un dev acostumbrado a REST y una sola API (Node, Nest, Spring, etc.) tiene que aprender GraphQL (esquema, resolvers, subgraphs) y a operar varios servicios. La curva de entrada es mayor sin necesidad real para este tipo de producto.
  • Dos (o más) despliegues, configuración de red entre gateway y microservicio, trazabilidad distribuida y secretos por servicio: todo eso tiene sentido con muchos equipos o muchos dominios; para una API interna con pocos dominios es sobrecarga.

Por qué la nueva arquitectura del back es mejor

  • Una sola API REST (NestJS): Un proceso, un puerto, un despliegue. El front consume esa API con los mismos endpoints y tipos. Sin gateway ni subgraphs; menos piezas que configurar y depurar.
  • Organización por dominio (libs api-*): Cada dominio (auth, usuarios, objetivos, fichajes, etc.) vive en su lib (api-auth, api-usuarios, …). Quien trabaja en “auth” va a una lib; no hay que repartir la lógica entre gateway, subgraph y microservicio.
  • Contrato común (shared-types): DTOs e interfaces compartidos entre front y back. Un cambio de contrato se hace en un sitio y ambos lados lo usan; menos errores y menos desalineación.
  • Escalable cuando haga falta: Si en el futuro un dominio crece o un equipo lo “posee”, esa lib se puede extraer a microservicio; la arquitectura actual (libs por dominio) ya está preparada para eso sin reescribir todo.

Resumen: el back antiguo (GraphQL + gateway + microservicio auth) no estaba alineado con el front y añadía complejidad operativa sin beneficio. El nuevo (una API NestJS + libs api-* + shared-types) es más simple de desarrollar, desplegar y entender, y sigue siendo común para cualquier dev (REST, tipos compartidos) y preparado para escalar si hace falta.

4. Qué es Nx, por qué y cómo funciona

Qué es Nx

Nx es un sistema de build y herramienta para monorepos (y proyectos grandes). No es un framework: trabaja junto a React, Next.js, NestJS, Angular, etc. Gestiona proyectos (apps y libs), sus dependencias, la caché de builds y la ejecución de tareas (build, test, lint, serve) de forma incremental.

Por qué Nx en este proyecto

  • Grafo de dependencias: Nx conoce qué libs usa cada app y cada lib. Con nx graph ves el grafo de proyectos y puedes hacer cumplir reglas (ej. “las features no se importan entre sí”).
  • Affected (solo lo tocado): nx affected -t build (o test, lint) ejecuta la tarea solo en los proyectos afectados por tus cambios. Menos tiempo en CI y en local.
  • Caché local y remota: Si algo no ha cambiado, Nx reutiliza el resultado del build/test anterior (local o, con Nx Cloud, entre máquinas). Builds y tests mucho más rápidos cuando la mayoría del código no cambia.
  • Boundaries y tags: Cada proyecto tiene tags en project.json (ej. type:feature, scope:auth). Con @nx/enforce-module-boundaries se prohíben imports no permitidos (ej. que una feature importe otra feature).
  • Generadores y consistencia: nx g @nx/js:library (u otros) generan libs con la misma estructura y configuración. Menos configuración manual y menos errores.

Cómo funciona (resumen)

  • nx.json en la raíz: configuración global (cache, task pipeline, plugins).
  • project.json (o project en package.json) por proyecto: nombre, sourceRoot, targets (build, test, lint, dev), tags.
  • Dependencias entre proyectos: Se derivan de los imports en el código (TypeScript/JavaScript). Nx construye el grafo leyendo esos imports.
  • Ejecución: nx run <proyecto>:<target> (ej. nx run front-biosstel:dev). nx run-many -t build --all para todos; nx affected -t build solo los afectados.

En resumen: Nx da estructura al monorepo, velocidad por caché y affected, y control por boundaries. Es el “motor” que hace que apps y libs encajen sin caos.

5. CI/CD y tests: núcleo del proyecto

CI/CD como núcleo inicial

El CI/CD no debe ser algo que se añade “cuando haya tiempo”: debe ser el núcleo inicial del proyecto. Antes de tener funcionalidades complejas, el pipeline debe estar funcionando (build, lint, tests) para que cada commit o cada PR se valide de forma automática.

  • Desde el primer día: Un pipeline que falle si el build o los tests fallan evita que se acumule deuda y que “en mi máquina funciona” se convierta en el estándar. Sin CI, es fácil acabar con merge directo a rama principal, sin revisión ni garantías.
  • Un solo pipeline en monorepo: Con Nx, el CI puede ejecutar solo lo afectado (nx affected -t build, nx affected -t test), lo que mantiene los tiempos de feedback bajos. El CD puede desplegar solo las apps o libs que hayan cambiado, según la estrategia del equipo.
  • Reglas de calidad en el pipeline: Lint, format, tests unitarios e integración (y opcionalmente E2E) como puerta de entrada. Si algo no pasa, no se mergea. Así la calidad no depende de que alguien “recuerde” ejecutar los tests.

Importancia de tests de integración y end-to-end (E2E)

  • Tests unitarios validan unidades aisladas (un componente, una función, un servicio). Son rápidos y útiles, pero no garantizan que las piezas encajen ni que el flujo completo funcione.
  • Tests de integración validan que varios módulos trabajen juntos (p. ej. un controlador + servicio + repositorio, o una feature de front que llama al API). Detectan desalineaciones de contrato, errores de configuración y fallos en la integración entre capas. Deberían ejecutarse en el CI (con API real o con mocks controlados).
  • Tests end-to-end (E2E) simulan al usuario: abren el front, navegan, rellenan formularios, comprueban que la respuesta sea la esperada. Cubren el flujo completo (navegador → front → API → BD o mocks) y son la última línea de defensa antes de producción. Herramientas como Playwright o Cypress encajan bien en un monorepo (tests E2E como target de la app front o de un proyecto dedicado).

Resumen: CI/CD desde el inicio y tests en varios niveles (unitarios, integración, E2E) forman el núcleo de un proyecto mantenible. Sin ellos, la velocidad a corto plazo se paga con bugs en producción y miedo a refactorizar; con ellos, el equipo puede iterar con confianza.

6. Arquitectura: apps + libs

El monorepo se organiza en apps (lo que se despliega) y libs (código reutilizable y por dominio).

CapasContenido
AppsFront (Next.js) y API (NestJS). Solo orquestan: importan shell/features en front y módulos api-* en back.
Shell (front)Layout, rutas, store global, providers. Punto de entrada de la SPA.
Features (front)Una lib por dominio: auth, usuarios, objetivos, fichajes, reportes, productos, inventario, alertas, empresa, operaciones. Cada una: data-access (acciones/efectos), componentes de página, hooks.
UI / UI-Layout (front)Componentes reutilizables (botones, inputs, cards) y layouts. Para que diseño trabaje sobre una lib estable.
Platform / Shared (front)Utilidades, hooks genéricos, configuración compartida.
Backend api-*Una lib por dominio (api-auth, api-usuarios, api-objetivos, etc.). Controladores, servicios, entidades. La app API los registra como módulos.
shared-typesDTOs, enums e interfaces compartidos entre front y back. Contrato por feature; la tecnología de cada feature puede ser distinta.

Lo importante es organizar por feature (dominio), no por tecnología: cada feature puede elegir su stack (back, front, base de datos) según necesidad; el contrato (shared-types o API) une las piezas.

Dependencias (resumido)

app front → shell → features features → ui | platform | shared front (features, ui, etc.) → shared-types app API → api-* libs api-* libs → shared-types

Así se mantiene una sola dirección: de apps hacia libs, y de features hacia UI/platform/shared y shared-types.

7. Cómo crear una feature (libs)

Para un nuevo dominio (ej. “pedidos”):

  1. Tipos compartidos: Añadir en shared-types los DTOs y enums que necesiten front y back (ej. Pedido, CreatePedidoDto).
  2. Backend: Crear lib api-pedidos (o el nombre del dominio): módulo NestJS con controlador, servicio y entidad. Exportar el módulo. Registrar el módulo en la app API.
  3. Frontend: Crear lib de feature pedidos bajo libs/frontend/features/:
    • data-access: acciones Redux, efectos (llamadas a API), selectores. Usar servicios que llamen a la API; no llamadas sueltas desde componentes.
    • Componentes de página y contenedores que usen useAppDispatch / useAppSelector y las acciones de la feature.
    • Registrar el slice en el store global (shell) y exponer rutas en el shell si aplica.
  4. Tags Nx: Marcar la lib con type:feature, scope:pedidos (o el nombre del dominio) para que las reglas de boundaries y el grafo sigan siendo correctos.

Regla de oro: La API de cada feature en front se usa solo desde su data-access (acciones/efectos). Los componentes no llaman al API directamente; disparan acciones.

8. Cómo escala la arquitectura

  • Más dominios: Añadir una lib de feature (front) y su api-* (back) no afecta al resto. Cada feature puede usar tecnología distinta (back en Python/Java/Node, BD distinta); el grafo y el CI siguen por feature.
  • Más apps: Distintas apps de front (Next.js, React Native, Vue, etc.) pueden consumir las mismas APIs de features. Cada app elige qué features usa; las APIs pueden estar en distintos lenguajes o servicios.
  • Más equipos: Los boundaries (tags Nx, reglas de import) permiten que un equipo “posea” un conjunto de libs (ej. auth + usuarios) y otro otro (fichajes, reportes), con integración en el shell y en la app API.
  • Escalabilidad por composición: No hace falta reescribir nada: se compone. Nueva vertical = nueva app + selección de libs existentes + libs nuevas solo para lo específico de esa vertical.

9. Productos verticales y libs como producto

Productos verticales

Un producto vertical es una aplicación concreta para un segmento (ej. solo fichajes + reportes para un cliente, o una white-label). Con esta arquitectura:

  • Creas una nueva app (front y/o back) que importa solo las libs que necesite ese producto.
  • Reutilizas shared-types, las api-* que toquen al dominio y las features de UI correspondientes. Cero duplicación de lógica.
  • Puedes tener una API “completa” para el producto principal y APIs o BFFs más pequeños para verticales concretos.

Vender o exponer libs (UI, API, módulos en general)

  • Libs de UI: La lib de componentes (botones, inputs, layouts, design tokens) se puede empaquetar y publicar (npm privado o público, con licencia). Otros productos o clientes pueden consumirla como dependencia. Diseño y marca quedan centralizados; el “producto” es la lib de UI.
  • Módulos de API: Cada lib api-* (auth, usuarios, fichajes, etc.) es un módulo con contrato claro (endpoints, DTOs en shared-types). Se pueden ofrecer como paquetes npm privados para que terceros monten su propia API registrando solo los módulos que quieran, o como base para SaaS por dominio (ej. “solo módulo de fichajes”).
  • En general: Cualquier lib bien acotada (shared-types, ui, api-auth, etc.) puede convertirse en producto: documentación, versionado semántico, changelog. La arquitectura ya está preparada porque cada lib es independiente y con dependencias explícitas.

10. Agnóstica en tecnología por feature (back, front, base de datos)

Cada feature es la unidad de organización. Lo que se comparte es el contrato (tipos, API), no la tecnología. Por tanto, cada feature puede ser agnóstica en tecnología tanto en back como en front.

  • Backend por feature: No hace falta que todas las APIs sean NestJS. Una feature puede estar en Node/NestJS, otra en Python (FastAPI, Django), otra en Java (Spring), etc., según lo que pida ese dominio (rendimiento, ecosistema, equipo). Lo importante es que expongan el mismo contrato (endpoints, DTOs definidos en shared-types o en una especificación compartida).
  • Bases de datos distintas por feature: Cada feature puede usar la base de datos que mejor le encaje: PostgreSQL para una, MongoDB para otra, Redis para caché de otra. Persistencia políglota por dominio, sin obligar a todo el sistema a una sola BD.
  • Frontends distintos por producto: Puedes tener una app Next.js para el producto principal, otra en React Native para móvil, otra en Vue o en otro framework para un vertical concreto. Cada app consume las APIs de las features que necesite; el contrato es común, la tecnología del front puede variar por app.
  • Organizar por feature, no por capa: La arquitectura gana porque el límite es el dominio (auth, fichajes, reportes…), no “todo el back en Java” o “todo el front en React”. Así se puede escalar, vender o extraer cada feature con independencia tecnológica.

Módulos federados: distintos frameworks en la misma app (migraciones y legacy)

Con módulos federados (Module Federation, p. ej. Webpack o Nx) puedes tener varios frameworks en la misma aplicación. Eso hace la arquitectura aún más flexible:

  • Migraciones graduales: En lugar de reescribir toda la app de golpe, se puede ir sustituyendo por partes: una ruta o un bloque en React, otro en Vue o Angular, conviviendo en la misma shell. Se migra feature a feature sin “big bang”.
  • Integrar proyectos legacy: Un front antiguo (jQuery, AngularJS, React clásico) se puede envolver como remoto federado y cargar dentro del shell nuevo. Poco a poco se van reemplazando pantallas o features por las nuevas libs; el legacy sigue funcionando hasta que se retire. Es la forma ordenada de traer proyectos heredados al monorepo y a la nueva arquitectura.
  • En general es mejor: Permite elegir el mejor momento y el mejor alcance para cada migración, reducir riesgo y repartir el esfuerzo en el tiempo. La app sigue siendo una sola desde el punto de vista del usuario; por dentro, cada módulo puede ser tecnología distinta.

Resumen: shared-types (o el contrato de API) es la fuente de verdad entre features. Que el back de auth sea NestJS, el de analytics Python y el de facturación Java es compatible con esta arquitectura; en front, los módulos federados permiten mezclar frameworks en la misma app y migrar o integrar legacy sin reescribir todo de una vez.

11. Libs de UI y diseño

Las libs de UI (ui, ui-layout) están pensadas para que diseño y desarrollo trabajen sobre el mismo sistema de componentes:

  • Componentes en una lib: Botones, inputs, cards, modales, etc. viven en la lib de UI. Diseño puede documentar (Storybook u otra herramienta) y definir variantes sin tocar la lógica de negocio.
  • Design tokens / temas: Colores, espaciados y tipografías se pueden centralizar (tokens o CSS variables) y consumir desde la lib de UI. Un cambio de tema se hace en un solo sitio.
  • Consistencia: Todas las features usan los mismos componentes. Menos duplicación de estilos y comportamientos; más coherencia visual entre productos verticales.
  • Colaboración diseño–dev: Diseño define en la lib de UI; desarrollo la importa en las features. Si hay una lib de iconos o de ilustraciones, también puede vivir como lib compartida.

Documentación de design tokens y Figma puede vivir en docs/ y referenciar la misma lib de UI que usa el código.

12. Por qué no microservicios desde el principio (auth, etc.)

Usar microservicios para todos los módulos (auth, usuarios, fichajes, etc.) desde el día uno suele ser contraproducente:

  • Coste operativo: Varios despliegues, varias bases de datos o esquemas, redes entre servicios, trazabilidad distribuida, configuración por servicio. Para un equipo pequeño o un solo producto, es sobrecarga sin beneficio real.
  • Complejidad antes de tiempo: API Gateway, service discovery, circuit breakers entre servicios, gestión de secretos y config por servicio solo tienen sentido cuando hay muchos servicios y equipos que los operan.
  • Una API ya escala: Una sola API NestJS que agrupa módulos (libs api-*) es más simple de desarrollar, desplegar y depurar. Escala bien en carga con réplicas y base de datos única hasta que el dominio o el equipo realmente pida separación.
  • Microservicios cuando aporten valor: Tiene sentido cuando un dominio crece mucho, un equipo lo “posee” y quiere desplegarlo aparte, o cuando hay requisitos de escalado/aislamiento muy distintos por dominio. No por defecto para “auth” o “usuarios” solo porque sean módulos.

Resumen: empezar con una API modular (libs) y extraer a microservicios solo donde y cuando compense. La arquitectura actual está pensada para eso.

13. Escalar a microservicios cuando haga falta: la arquitectura es un Lego

Esta arquitectura permite escalar a microservicios sin reescribir: cada lib es una pieza que se puede sacar o dejar dentro.

  • Cada api-* es un bounded context: api-auth, api-usuarios, api-fichajes, etc. ya están acotados por dominio. Para convertirlos en microservicio: se extrae la lib a un proyecto/servicio que exponga HTTP (o gRPC), se despliega aparte y se pone delante un API Gateway que enrute (ej. /auth/* → servicio auth). El contrato (shared-types) no cambia; el cliente sigue llamando al mismo gateway.
  • Migración gradual: Puedes sacar primero solo uno (p. ej. auth) a un microservicio y dejar el resto en la API monolítica. No hace falta “pasarse a microservicios” de golpe. Donde quieras, cuando quieras.
  • Lego: Las piezas (libs/features) encajan por contrato (tipos, endpoints). Pueden vivir dentro de una sola API o en servicios separados; la frontera es de despliegue y operación, no de código. Incluso puedes reescribir una pieza en otro lenguaje (Python, Java…) al extraerla; lo que une todo es el contrato, no la tecnología.

Conclusión: La arquitectura no te cierra la puerta a microservicios ni a tecnologías distintas por feature. Es un Lego: mismo contrato por feature; cada pieza puede ser NestJS, Python, Java, otra BD u otra app front según necesidad. Añadir API Gateway, Config Server o circuit breakers tiene sentido cuando ya existan varios servicios (ver documentación de microservicios en esta carpeta).

14. Flujo de trabajo recomendado

  • Nueva funcionalidad: Definir tipos en shared-types → implementar o extender api-* en back → implementar data-access y pantallas en la feature de front → registrar en shell (store, rutas).
  • Cambio solo en UI: Modificar componentes en ui o ui-layout; las features que los usan se benefician sin tocar lógica.
  • Nuevo producto vertical: Nueva app que importe shell (o un shell ligero) y solo las features necesarias; misma API y shared-types.
  • Build y tests: Desde la raíz, construir y testear por proyecto (Nx solo afecta lo cambiado). Revisar nx graph para ver dependencias antes de refactors grandes.

15. Servidor de mocks (no depender de datos ni del back)

Para no crear mocks aleatorios en código ni estar bloqueado cuando el backend o los datos no están listos, conviene usar un servidor de mocks que simule la API con respuestas coherentes y reutilizables.

Por qué usar un mock server

  • El front puede desarrollarse y probarse sin depender de la API real ni de la base de datos.
  • Las respuestas son consistentes (mismo contrato que la API real) y versionables (en archivos o imágenes Docker), no datos “random” repartidos por el código.
  • Útil en CI, demos y para QA cuando no se quiere levantar todos los servicios.

Opciones y cómo usarlas

HerramientaUso típicoCómo instalar / arrancar
json-server API REST simple a partir de un JSON. Muy rápido para prototipar y para CRUD. pnpm add -D json-server. Crear un db.json con las entidades (ej. {"users": [], "objetivos": []}). Arrancar: npx json-server --watch db.json --port 3001. Opcional: --routes routes.json para rutas custom.
MockServer Simula HTTP/HTTPS con expectativas (request → response). Muy flexible, ideal para contratos complejos y tests E2E. Docker: docker run -d -p 1080:1080 mockserver/mockserver. Configurar expectativas vía API REST en localhost:1080 o con archivos. Cliente Node: pnpm add -D mockserver-client para definir mocks desde tests.
MSW (Mock Service Worker) Intercepta peticiones en el navegador (o en Node). No requiere servidor aparte; los mocks viven en el proyecto (handlers). pnpm add -D msw. Definir handlers (GET/POST por path) y usar setupServer(...) en tests o en desarrollo. Bueno para tests unitarios e integración sin levantar backend.

Recomendación rápida

  • Desarrollo local sin API: json-server con un db.json alineado con shared-types (mismos nombres y estructura). El front apunta a http://localhost:3001 (o el puerto que uses) cuando se usa el mock.
  • Tests E2E o contratos estrictos: MockServer en Docker; las expectativas se cargan al arrancar y devuelven siempre las mismas respuestas.
  • Tests unitarios / integración en front: MSW para no tocar la red real; los mismos handlers pueden usarse en Jest/Vitest y en dev.

Tener un único “contrato” de mocks (archivos o imagen Docker) evita mocks dispersos y aleatorios en el código y desbloquea al equipo cuando faltan datos o el back no está disponible.
