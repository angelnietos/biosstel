Comparativa: Proyecto inicial vs Monorepo actual

Biosstel — Documento objetivo para CEO, CTO y Negocio

Proyecto inicial · Dos repositorios (front + API) Solución actual · Un solo repositorio (monorepo)

Objetivo del documento

Comparar de forma objetiva la solución inicial (dos repositorios: interfaz y servidor) con la solución actual (monorepo único), para apoyar decisiones de inversión, organización del equipo y evolución del producto. Las conclusiones se basan en el análisis del código de ambas soluciones.

1. Resumen ejecutivo

En una frase: La solución actual (monorepo) mejora en organización, alineación front–back, tipos compartidos, un solo flujo de trabajo y base para escalar. La solución inicial era más “simple” de entender al primer vistazo pero con mayor riesgo de desincronización y coste de coordinación.

Conclusión: el monorepo actual gana en todos los criterios excepto en la “simplicidad” inicial; esa ventaja es efímera frente a los beneficios a medio plazo.

2. Para el CEO

Mensaje en un minuto

  • Antes: Dos proyectos independientes (interfaz y servidor), sin definición única de datos, con tecnología en el servidor (GraphQL) que la interfaz no utilizaba. Dos sitios que mantener y coordinar.
  • Ahora: Un solo proyecto (monorepo): misma interfaz y mismo servidor, datos y reglas definidos una vez y compartidos. Una única forma de desplegar, probar y documentar.
  • ¿Está mejor? Sí: menos duplicación, menos riesgo de desincronización, más rapidez para añadir funcionalidades y más facilidad para el equipo y para incorporar personas.
  • Único punto a favor del pasado: Los proyectos antiguos parecían más simples al primer vistazo. A cambio, eran más frágiles y difíciles de escalar.

Recomendación: Mantener el monorepo actual. No hay motivo para volver atrás ni para añadir complejidad (p. ej. GraphQL) que no aporte valor en este producto.

3. Para el CTO

Hallazgos del análisis en profundidad (proyecto inicial)

  • Contrato login: La interfaz enviaba el campo username en la mutación GraphQL de login; el esquema del servidor y el resolver esperan email. Incompatibilidad directa entre front y back.
  • Puertos por defecto: La interfaz apuntaba por defecto al puerto 3001; el gateway GraphQL corría en 8001 y el microservicio de auth en 5001. Sin configurar variables de entorno, las peticiones no alcanzaban el backend.
  • CORS: Tanto el gateway como el microservicio de auth tenían origin: "*", riesgo de seguridad en APIs con autenticación.
  • JWT: El secreto por defecto era cadena vacía si no se configuraban variables de entorno.
  • Inicialización duplicada: En el servidor del microservicio auth, middlewares y rutas se registraban dos veces (en el constructor y en listen).
  • Datos mock: Múltiples listas de datos estáticos (usuarios, tareas, objetivos, departamentos, etc.) para desarrollo; la interfaz dependía de ellos en varias pantallas.

4. Para Negocio / Producto

Impacto en el producto y en la organización

  • Velocidad de entrega: En el monorepo, añadir una funcionalidad implica un solo repo, tipos compartidos y tests unificados; menos coordinación entre equipos front y back.
  • Calidad y consistencia: Una sola definición de datos reduce errores (p. ej. pantallas que esperan campos que la API no devuelve) y retrabajos.
  • Onboarding: El proyecto inicial podía parecer “más fácil” al principio (menos carpetas); el monorepo está mejor documentado y es más predecible a medio plazo.
  • Riesgo operativo: En el inicial, desplegar front y API por separado con versiones desalineadas aumentaba el riesgo de incidencias; en el actual, un único flujo lo reduce.

7. Métricas y criterios (objetivos)

Criterios usados para la comparación:

  • Un solo lugar de verdad: ¿Hay una única definición de datos entre front y back? (Inicial: no; Actual: sí.)
  • Alineación de contrato: ¿El front usa lo que el back expone? (Inicial: no; Actual: sí.)
  • Operación unificada: ¿Un clon, una instalación, un pipeline? (Inicial: no; Actual: sí.)
  • Escalabilidad de dominio: ¿Añadir una nueva área de negocio es modular y claro? (Inicial: limitado; Actual: sí.)
  • Onboarding: ¿Alguien nuevo entiende rápido? (Inicial: aparentemente sí; Actual: requiere documentación; a medio plazo el actual es más predecible.)

La comparativa se centra en estructura, mantenibilidad y riesgo operativo; no en rendimiento ni coste exacto de migración.

8. Recomendación

Mantener la solución actual (monorepo).

  • Mejor en organización, consistencia de datos, productividad del equipo, escalabilidad, mantenimiento y despliegue.
  • API coherente (REST) sin tecnología (GraphQL) que no se usaba.
  • El proyecto inicial solo gana en “simplicidad” inicial; esa ventaja es efímera frente a los beneficios del monorepo a medio y largo plazo.

No se recomienda volver a la estructura de dos repos ni incorporar GraphQL sin una necesidad explícita; las decisiones están documentadas en la documentación de soporte.
